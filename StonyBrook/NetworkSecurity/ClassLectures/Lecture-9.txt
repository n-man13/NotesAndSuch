Problem
▶Consider a prover and a verifier, where the prover has a large
set of data and the verifier wants to check if an element is in
the set.▶Attempt 1: Send the entire set? Inefficient!
▶Attempt 2: Send the list of hashes of the elements, and ask
the verifier to look up the table?
▶Verifiers should not need to download the entire list.
▶Key idea: Use a tree structure to represent the data, where
the leaves are the data, every inner non-leaf stores the hash of
its children, and send the path from the element back to the
root.

Merkle Trees Append-Only Merkle Trees
Append-Only Merkle Trees
▶We can also construct an append-only merkle tree
▶In an append-only merkle tree, we can only add new leaves to
the tree, but cannot remove or modify existing leaves.
▶When a tree is modified, we have to recompute the hashes
from the modified leaf all the way up to the root.
▶How to ensure that the tree contains all the leaves that were
previously added?▶We need a consistency proof that given two merkle trees T1
and T2, where T1is a subtree of T2, we can prove that T2
contains all the leaves of T1.

ACME ACME
ACME
▶Automated Certificate Management Environment▶Proposed in RFC 8555
▶Using JSON messages over HTTPS to communicate between
the CAs and the client’s servers that deploy the certs (most
likely web servers)
▶Accounts are bound to a public key, which is used to sign
requests to the ACME server
▶Challenge-Response protocol to prove control over a domain.
For example:
▶HTTP: the client places a file with a specific content at a
specified location
▶DNS: the client creates a TXT record in the DNS zone for the
domain with a specific string
▶Once the challenge is completed, the ACME server will issue a
certificate for the domain.

ACME: Domain Validation
in Detail

ACME   Domain Validation
Overall ACME Protocol Flow
15 / 351 Client generates an account key pair and registers with the ACME server
2 Client submits an order for a domain (e.g. example.com)
3 Server responds with an authorization object containing challenge options
4 Client fulfils a challenge (HTTP -01 or DNS -01) and signals readiness
5 Server verifies the challenge — authorization marked valid
6 Client submits a CSR signed with the domain key; server issues the certificate
All requests are JWS -signed with the account private key; each carries a fresh server -issued nonce → replay attacks are prevented.

ACME   Domain Validation
Key Authorization & JWK Thumbprint
16 / 35► Every challenge response embeds a key authorization string
▸ key_authorization  =  token  ||  "."  ||  base 64url(SHA -256(account_public_key_JWK))
► JWK = JSON Web Key (RFC 7517 ) — a JSON serialization of the account public key
▸ { "crv":"P -256", "kty":"EC", "x":" …", "y":" …" }  (keys in lexicographic order, no whitespace)
► The JWK Thumbprint (RFC 7638 ) is SHA -256 of that canonical JSON, base 64url-encoded
▸ Result is always 43 characters — compact enough for DNS TXT records
► Security role: binds the challenge to a specific ACME account
▸ An attacker who intercepts a token cannot reuse it — the thumbprint of their key won't match what the server 
expects
► DNS -01 hashes the key authorization a second time before placing it in DNS
▸ dns_value  =  base 64url(SHA -256(key_authorization))

HTTP -01 Challenge — Setup
17 / 35► Server issues a random token (URL -safe base 64)
► Client computes the key authorization string and serves it at:
▸ http://<domain>/.well -known/acme -challenge/<token>
► Must be plain HTTP (port 80), Content -Type: text/plain
► Client POSTs a signed 'ready' message to the ACME server
► Server fetches the URL and verifies:
▸ ① token matches  ② thumbprint matches account keyACME Client

ACME   HTTP -01 Challenge
HTTP -01 Challenge — Security Properties
18 / 35► Proof of domain control
▸ Only the operator of the web server at that domain can place the file at the exact URL
► Replay prevention
▸ Nonce in each signed ACME request; token is single -use and short -lived
► Token -forwarding attack prevented
▸ Key authorization binds token to the requesting account's public key thumbprint
▸ Intercepted token is useless without the matching private key
► BGP hijacking: partially mitigated by multi -perspective validation
▸ Let's Encrypt queries from multiple geographically diverse vantage points
▸ Attacker must hijack the victim's IP prefix simultaneously on all paths — much harder
► Limitations
▸ Requires port 80 open and a running web server  •  Cannot issue wildcard certificates

ACME   DNS -01 Challenge
DNS -01 Challenge — Setup
19 / 35► Server again issues a random token
► Client derives the DNS value (double -hashed):
▸ key_auth   = token + "." + base 64url(SHA -256(JWK))
▸ dns_value  = base 64url(SHA -256(key_auth))
► Client creates a DNS TXT record:
▸ _acme -challenge.<domain>   IN   TXT   "<dns_value>"
► Server queries multiple DNS resolvers and verifies the TXT record
► Double -hashing rationale: DNS TXT length limits; hides the account key 
thumbprint from public DNS
 Delegation Trick
CNAME _acme -challenge to a separate zone managed by a 
cert-management service. Primary DNS API credentials stay 
isolated — only the delegated zone needs write access.
  Wildcard Support
DNS -01 is the only challenge type that supports wildcard 
certificates (*.example.com).

ACME   DNS -01 Challenge
DNS -01 Challenge — Security Properties
20 / 35► Stronger proof of domain control than HTTP -01
▸ Requires write access to the DNS zone — not just a running web server
► Works behind NAT or firewalls — no inbound HTTP port required
► BGP hijacking is harder to exploit
▸ DNS queries fan out to many resolvers globally; no single HTTP endpoint to intercept
▸ Attacker would need to poison resolvers across many independent networks simultaneously
► New risk: DNS provider API key exposure
▸ Automation requires credentials to the DNS provider API
▸ Mitigation: scope API key to _acme -challenge subdomain only (if provider supports it)
▸ Or delegate _acme -challenge via CNAME to an isolated zone (see previous slide)
► Account -binding: same key authorization mechanism prevents token forwarding

Certificate Transparency Introduction
Certificate Transparency
▶Certificate Transparency (CT) is a system that provides a
public log of allcertificates issued by public CAs
▶Proposed by Google
▶The goal of CT is to detect and prevent the issuance of
fraudulent certificates by CAs
▶When a CA issues a certificate or when a party obtains a
certificate, they submit the certificate to one or more CT logs
▶The log operators may also misbehave, so we need watchers
that watches the watchers
▶The log is an append-only Merkle tree
▶The head is signed, and is called Signed Tree Head (STH).

Certificate Transparency CT Logs
Signed Certificate Timestamps (SCTs)
▶When a party submits a certificate to a CT log, the log will
not immediately append the certificate to the log
▶Instead, the log merely promises to include the certificate in
the log within the maximum merge delay(MMD), usually 24
hours
▶The log will generate a Signed Certificate Timestamp (SCT)
for the certificate as a promise
▶the SCT includes, among other things,
1. the entry id
2. a timestamp at which the certificate is accepted, and
3. a signature of the submitted certificate signed by the public
key of the log server.

Certificate Transparency SCTs and Inclusion Proofs
Meaning of SCTs
▶What does it mean when a certificate has SCT and is checked
to be valid?▶It only means that the certificate was submitted to a CT log
▶It also does not mean that the certificate is valid or
trustworthy!
▶Notice that a malicious CA can still issue a fraudulent
certificate and submit it to a CT log, and obtain an SCT for it.
▶It just means that the owner of the certificate would be able to
detect the fraudulent certificate, if they were indeed
monitoring the logs for their domain. If they do detect it, they
can try to take action like revocation.
▶It does not mean that the certificate was actually included in
the log yet!

Certificate Transparency Parties Involved
Monitors
▶Monitors are third-party services that monitor CT logs
▶They have a list of domains to monitor, and will periodically
check the CT logs for new entries
▶When a new certificate is found, the monitor will notify the
domain owner about the new certificate.
▶They should also fetch all the entries in the log to make sure
that the root hash is consistent and the signature is valid in
the STHs
▶They would either keep a copy of the log locally and compute
the new root hash, or rely on the consistency proofs provided
by the log server to track the incremental changes in the log.
