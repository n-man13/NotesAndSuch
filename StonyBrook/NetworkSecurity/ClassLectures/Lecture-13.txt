Information Flow
Information Flow
▶A concept in Multi-level Security (MLS)
▶In a program or system, we want to reason about how the
“information” flows from varialbes to variables.
▶Security labels
▶Simplest case: High and Low
▶High = Highly sensitive, Low = Low sensitivity
▶A program takes in inputs with low and high sensitivity, return
things with low and high sensitivity
▶Assume: high inputs are secret, low inputs are public; high
outputs are secret, low outputs are public
▶If a user input (low) flows to commands in the server (high
output), then bad things may happen.
▶E.g., SQL injection, if user input is used to construct a SQL
query without sanitization

Information Flow Noninterference
Noninterference
▶Model information flow with inputs and outputs
▶The idea that the high inputs should not affect, or flow to, the
low outputs.
▶If a program fail to satisfy noninterference, then in a sense,
the high inputs are leaked into the low outputs.
▶If a program does satisfy noninterference, then given the
output, one learns nothing about the high inputs.

Information Flow Noninterference
Noninterference
▶More formally, letPbe a deterministic program, andI HandI L
be the high and low inputs respectively, andO HandO Lbe the
high and low outputs respectively, soP(I H,IL) = (O H,OL).
▶Then we sayPhas thenoninterferenceproperty if for any
IH1andI H2and anyI L, if we have(
P(IH1,IL) = (O H1,OL1)
P(IH2,IL) = (O H2,OL2), thenO L1=O L2.
▶We also say that there is no information flow betweenI Hand
OL.
▶Notice that the property has to quantify over two traces of
the program, so noninterference is also called ahyperproperty.

Information Flow Non-deducibility
Non-deducibility
▶Similar to non-interference, but instead of learning nothing
about the inputs from the outputs, we do not want to be able
to eliminate (high) input from the set of possible input.
▶Works for non-deterministic programs
▶For any outputO Lin the domain ofPand for allI H, there
exists a low inputI Lsuch that the low output ofP(I H,IL) is
OL.
▶The negation would be that there exist someO Lsuch that we
haveI Hwhere noI Lcan be used to produceO L.
▶This means that after observing the low outputO L, we can
rule out the high inputI H.

Information Flow Covert Channels
Covert Channels
▶In the functional viewpoint, we care about the pure output of
the program.
▶However, low outputs should include everything that can be
observed by the user.
▶Consider the program
int f(high int x, low int y) {
if (x > 0) {
usleep(1000);
}
return y;
}
▶We can deduce the value ofxfrom the timing of the output!

Side-Channel Attacks

▶Sometimes the violation is unintentional
▶We call these violationsside-channel attacks
▶Side-channel attacks can be timing attacks, power analysis
attacks, etc.

CBC Padding Oracle Attack Introduction
Introduction
Now we shift our focus to a real-world attack on CBC. There is an
attack that allows an attacker to decrypt ciphertexts without
knowing the key. Attempts to fix the attack has lead to a
side-channel variant of the attack.

CBC Padding Oracle Attack Introduction
CBC Padding Oracle Attack
▶The attack is based on AES with CBC mode of operation
▶Recall that CBC mode is malleable (What does it mean?)
▶Ci=E k(Pi⊕C i−1) (whereC 0is the IV)
▶Pi=D k(C1)⊕C i−1
▶Since AES only works on blocks of 16 bytes, we need to pad
the plaintext to a multiple of 16 bytes.
▶This is typically done with PKCS#7 padding.

CBC Padding Oracle Attack PKCS#7 Padding
Padding Removal
▶Notice that while every string can be padded, not every string
with length being a multiple of 16 corresponds to a valid
unpadded plaintext.▶The unpad function would have to return an error if the
padding is invalid.
▶Thus, after decrypting the ciphertext, it is natural to check for
the padding, and return an error if the padding is invalid.
▶However, this leads to an attack where we can decrypt the
ciphertext without knowing the key.
▶Attributed to Vaudenay in 2002.
▶A kind of adaptive chosen-ciphertext attack.

CBC Padding Oracle Attack PKCS#7 Padding
Padding Oracle
▶Abstractly, let us assume that there is a black-box functionO
that takes a ciphertextCand returns whether the padding is
valid.
▶We will call this function apadding oracle.
▶In practice, we may submit a ciphertext to a web server. If the
padding is valid, the server may proceed with the decrypted
data. Otherwise, the server may return an error message.
▶This produces a one-bit signal that tells us whether the
padding is valid.

CBC Padding Oracle Attack Vaudenay’s Attack
Vaudenay’s Attack
▶Assume we have a ciphertextC= (IV,C), and we know that
the padding is valid, and that the padding length isp.
▶Then we know that the lastpbytes of the plaintext all have
valuep.
▶Now we can maulIV, so that the lastpbytes ofChas value
p+ 1.▶We can send the modified ciphertext to the oracle, and check
if the padding is valid.
▶If yes, then we know that byte before the lastpbytes of the
plaintext has valuep+ 1
▶If the padding is invalid, then maul the byte before the lastp
bytes of the IV until the padding is valid.

CBC Padding Oracle Attack Vaudenay’s Attack
Analysis
▶We only need at most 256 queries to the oracle to recover one
byte of the plaintext.
▶We can repeat this for every byte of the plaintext block.
▶We can recover the entire plaintext blockPin at most
256×16 = 4096 queries.

Recovering the original padding length
▶Recovering the original padding length is easy.
▶Assume the original ciphertext (IV,C) has correct padding

CBC Padding Oracle Attack Vaudenay’s Attack
Corollary
This shows that AES-CBC with PKCS#7 padding will fail to have
a kind of IND-CPA security where the adversary also has access to
a padding oracle (in addition to the encryption oracle).
Exercise
Show that we can generalize this to recover a three-block
ciphertext (IV,C 1,C2).
Exercise (Encryption Oracle (hard))
Show that besides recovering ciphertexts, we can actually encrypt
arbitrary plaintexts with the padding oracle without knowing the
key. (Hint: if you know the plaintext corresponding to some
ciphertexts, then you can maul that ciphertext to correspond to
any plaintext you want.)

CBC Padding Oracle Attack Side-Channel Attack
TLS
▶Consider TLS 1.0 to 1.2, where if AES-CBC is chosen as the
ciphersuite, then the record protocol will perform a
MAC-Encode-Encrypt operation to encrypt the plaintexts.▶The MAC is first computed, the plaintext is then padded and
encrypted.
▶If the padding is invalid, then the server will return an alert
message without computing MAC.
▶If an attacker messed with the non-padding part, unpad would
succeed, but MAC would still fail, returning another
(encrypted) alert message.
▶A timing difference between valid and invalid padding!
Brice Canvel, Alain Hiltgen, Serge Vaudenay, and Martin
Vuagnoux. Password interception in a SSL/TLS channel.
(CRYPTO 2003)
