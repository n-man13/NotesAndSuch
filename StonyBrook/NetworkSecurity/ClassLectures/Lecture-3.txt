Random One -Time Pad
•Fix the vulnerability of classic ciphers by using very long keys 
• The key to be used is a random string as long as the 
message 
• Encryption is similar to shift ciphers 
• Invented by Vernam  in the 1920s

Random One -Time Pad
• Let Zm = {0,1,2,…,m -1} be the alphabet 
• Plaintext space = Ciphertext space = Key space = ( Zm)n 
• The key is chosen uniformly randomly
• Plaintext     P  = (p1 p2 … pn)
• Key              K  = (k1 k2 … kn) 
• Ciphertext C  = (c1 c2 … cn)
• EK(P)   = (p1+k1 p2+k2 … pn+kn) mod m [ Encryption ]
• DK(C)  = (c1-k1 c2-k2   … cn-kn) mod m    [ Decryption ]

Exclusive OR ( ⨁)
⨁ 1 0
1 0 1
0 1 0
• A ⨁ A = 0 
• A ⨁ 1 = ~A (~ denotes negation or NOT)
• A ⨁ 0 = A 
• A ⨁ A ⨁ B = B

Binary Version of Random OTP
• Let Zm = {0,1} be the alphabet 
• Plaintext space = Ciphertext space = Key space = {0,1}n 
• The key is chosen uniformly randomly
• Plaintext     P  = (p1 p2 … pn)
• Key              K  = (k1 k2 … kn) 
• Ciphertext C  = (c1 c2 … cn)
• EK(P)   = (p1 ⨁ k1 p2 ⨁ k2  … pn ⨁ kn) [Encryption ]
• DK(C)  = (c1 ⨁ k1 c2 ⨁ k2   … cn ⨁ kn) [Decryption ]

Which attacks to defend against?
• It is very tempting to always design a cryptosystem that is 
secure against the strongest attacks 
• Cryptosystems that protect against the strongest attacks are 
often less efficient 
• While designing a cryptosystem, to decide which attacks to 
protect against, one has to consider the application at hand 
• For the application in question, if only the weaker attacks 
make sense, then cryptosystems should be designed 
accordingly

Stream Ciphers
• Modern stream ciphers approximate random one -time pad 
• Recall, in random one -time pad, the key is a random  string of 
length at least the same as the message 
• Stream ciphers : 
• Insight : replace “ random ” by “ pseudo -random ”  
• Use pseudo -random number generators 
• PRNG: {0,1}s → {0,1}n
•Expand a short (e.g., 128 -bit) random seed into a long (typically unbounded) 
string that “looks random”
• Secret symmetric key is the seed
• Basic encryption method: EK(M) = M ⨁ PRNG(K)

RC4 Stream Cipher
•A proprietary cipher owned by RSA, designed by Ron Rivest in 1987
• Became public in 1994 
• Simple and effective design 
• Variable key size (typically 40 to 256 bits) 
• Output length unbounded 
• Used to widely applied (web SSL/TLS, wireless WEP)
• Extensively studied but not quiet a completely secure PRNG
• The first part of output is biased when used in a stream cipher 
• Should use RC4-Drop[n]
• This drops the first n bytes before using the output 
• Conservatively, set n = 3072

Pseudorandom Number Generators
•Security of stream ciphers fully rely on the underlying PRNG to be 
cryptographically secure or strong
• Useful for cryptography, simulation, randomized algorithm
• The same seed always gives the same output stream
• Simulation requires uniformly distributed sequences
• Cryptographically secure PRNG requires unpredictable sequences 
• Security of PRNGs : Secure PRNGs should satisfy the “ next -bit test ” 
•Next -bit Test : Given consecutive sequence of bits output (but not the seed), the 
next bit should be hard to predict
• Some PRNGs are weak : knowing output sequence of sufficient length, 
can recover key 
• Do not use such PRNGs for cryptographic purposes 

Stream Cipher Security
• Under known plaintext, chosen plaintext, or chosen ciphertext, the adversary 
knows the key stream (i.e., PRNG(key))
• Security of stream ciphers mainly rely on the strength of the underlying PRNG
• PRNG must be “unpredictable” 
• An adversary cannot computationally distinguish input from a complete random number 
generator and a PRNG 
• Precise definition can be found in the “Modern Cryptography” book ( I will share the 
necessary portion) 
• If the same key is used twice, then it is easy to break 
• This is a fundamental weakness of stream ciphers
• It exists even if the underlying PRNG used in the stream cipher satisfies the next -bit test

Stream Cipher in Practice
• In practice, one key is used to encrypt many messages 
• Example : wireless communication 
• Solution : Use an initialization vector IV (sometimes, called nonce) 
• EK(M) := [IV, M ⨁ PRNG(K || IV)]  {|| refers to concatenation}
• Expectations from the IV
• IV is sent in clear to the receiver 
• IV needs integrity protection but not confidentiality 
• IV ensures that key streams do not repeat, but does not increase the cost of 
brute -force attacks 
• Without key, knowing IV will not let the attack decrypt the ciphertext 
• One has to ensure that the IV never repeats! Why? How? 

Deterministic and Randomized
• Encryption can be randomized 
• Same message, same key, running the same algorithm twice results in two 
different ciphertexts 
• Enck(m) = (r, PRNG(k || r) ⨁ m ) – the ciphertext includes two parts, a 
randomly generated r, and a second part 
• Decryption is always deterministic in the sense that
• For the same ciphertext and the same key, running the decryption 
algorithm twice always results in the same plaintext  
•Each key induces a one -to-many mapping from plaintext space to the 
ciphertext space
•Corollary : Ciphertext space must be equal to or larger than 
the plaintext space

Deterministic Encryption
• Deterministic encryptions are bad and insecure in most of the cases 
• This will be formally explained later 
• One can use frequency analysis if they were to know the underlying 
distribution of the plaintext space 
• An example paper of such an attack on deterministic encryption: 
•Inference attacks on Property -preserving Encrypted Databases 
•Published in ACM CCS 2015
•The paper can be found here: https:// cs.brown.edu /~seny /pubs/ edb.pdf

Block Ciphers
• A n-bit plaintext is encrypted to an n -bit ciphertext  
• P : {0,1}n
• C : {0,1}n
• K : {0,1}n
• Enc: K x P → C 
• Enck(.) can be viewed as a permutation of {0,1}n
• Dec: K x C → P
• Deck(.) can be viewed as Enck-1(.) 
• Block size = n 
• Key size = s 

Data Encryption Standard (DES)
• Designed by IBM, with modifications proposed by NSA 
• US national standard from 1977 to 2001 
• Block size 64 bits 
• Key size 56 bits 
• Has 16 rounds 
• Designed mostly for hardware implementations 
• Software implementations are slow 
• Considered insecure now 
• Vulnerable to brute -force attacks 
• Triple DES:  
• EK3DK2Ek1(M) has 112 -bit strength, but slow

Advanced Encryption Standard (AES)
• From 1999, AES replaced DES as the standard for block ciphers 
• Aka Rijndael (invented by Joan Daemen and Vincent Rijmen ) 
• Designed to be efficient in both hardware and software across a 
variety of platforms 
• Block size: 128 bits  
• Variable key size: 128, 192, or 256 bits  
• No known exploitable algorithmic weaknesses
• Implementation may be vulnerable to timing attacks (largely due to 
CPU’s architecture of using caches) 
• Intel now has AES -NI, CPU -based implementation for AES 
• Timing -based side channel attacks are no longer an issue 

Encryption Modes
• A block cipher encrypts only one block of message
• In case of AES, the block size is 128 bits 
• What if the message being encrypted is longer than the 
block size?  
• Need a way to extend “block ciphers” to encrypt an 
arbitrarily long message 
• We want to ensure that if the underlying block cipher is 
secure, then the encryption mode is secure 
• The main aim is to provide semantic security assuming 
the underlying block ciphers are strong
• Will introduce “semantic security” in a bit

ECB
• The easiest and the most intuitive one out of the different 
modes of AES 
• Recall, AES encrypts one block of message only 
• Divide up the message to be encrypted into 128 -bit blocks 
(pad the message when the message length is not a 
multiple of 128) 
• Each block is then encrypted separately 
• Assume there are s number of blocks in the message
• Encryption:  ∀0≤𝑖<𝑠: Ci = EK(Pi) 
• Decryption:  ∀0≤𝑖<𝑠: Pi = DK(Ci)

ECB
• ECB is a deterministic encryption scheme
• The same plaintext data block results in the same ciphertext 
block 
• Reveals patterns of data when a data block repeats 
• When the same key is used, the same message is encrypted in 
the same way 
• Usage:  Not recommended to encrypt more than one block of 
data

CTR
• Gives a stream cipher from a block cipher 
•Randomized encryption : Repeated text gets mapped to different 
encrypted data
• IV or counter must be randomly chosen to get this benefit 
•Random access :
•Encryption and decryption of a block can be done in random order
• This is very useful for hard -disk encryption 
• When one block changes, re -encryption only needs to encrypt that 
block. In case of CBC, all later blocks also need to re -encrypted

Insight of IND -CPA
•Perfect secrecy means that any plaintext is encrypted to a 
given ciphertext with the same probability
•Given any pair of M0 and M1, the probabilities that they are 
encrypted into a ciphertext C are the same
•Hence, no adversary ( with no restrictions on computing power ) can 
tell whether C is ciphertext of M0 or M1.
•IND-CPA means
•With bounded computational resources , the adversary cannot tell 
which of M0 and M1 is encrypted in C

Comparison
•If a cipher has only computational security, then it can be 
broken by a brute force attack (e.g., enumerating all 
possible keys)
•Weak algorithms can be broken with much less time
•How to prove computational security?
•Assume that some problems are hard (requires a lot of 
computational resources to solve), then show that breaking 
security means solving the problem

Security Requirements
Given a function h:X →Y , then we say that h is:
•Preimage resistant (one -way):
    if given y Y it is computationally infeasible to find a value x X s.t. 
h(x) = y
•2nd preimage resistant (weak collision resistant):
    if given x  X it is computationally infeasible to find a value x’  X, 
s.t. x’x and h(x’) = h(x)
•Collision resistant (strong collision resistant):
    if it is computationally infeasible to find two distinct values x’,x  X, 
s.t. h(x’) = h(x)

Length of Hash Outputs
•The Weakest Link Principle: 
•A system is only as secure as its weakest link .
•Hence all links in a system should have similar levels of security.
•Because of the birthday attack, the length of hash outputs in general 
should double the key length of block ciphers 
•SHA-224 matches the 112 -bit strength of triple -DES (encryption 3 times 
using DES)
•SHA-256, SHA -384, SHA -512 match the new key lengths (128,192,256) in 
AES
•If small output size is highly important, and one is sure that collision -
resistance is not needed (only one -wayness  is needed), then same size 
should be okay.

•A MAC scheme is a hash family, used for message authentication
